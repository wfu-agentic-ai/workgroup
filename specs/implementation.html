Workgroup Implementation Details

Workgroup: WFU Agentic AI Workgroup Status: Active Last Updated: 2026-02-06

Pedagogical framework

Learning theory foundation

  • Primary framework: Social constructivism — participants learn by doing, collaborating, and building on each other’s knowledge
  • Instructional model: Task-driven learning with worked examples
  • Key principles: Progressive scaffolding, peer support, authentic tasks
  • Rationale: CLI skills are best learned through hands-on practice with real tasks, not lectures

Design philosophy

  • Participant-centered: Content responds to what participants need, not a fixed syllabus
  • Low barrier to entry: No prerequisites, no grades, no penalty for missing sessions
  • Modular content: Each tutorial stands alone; participants can skip ahead or circle back
  • Async-first: All content is available on the website; meetings supplement, not replace

Content structure

Topic organization

Foundation
├── CLI basics (terminal, commands, file operations)
├── Working with paths (absolute, relative, home)
└── Shell configuration (dotfiles, PATH, aliases)

Version control
├── Git fundamentals (init, add, commit, log, diff)
└── GitHub collaboration (remotes, push/pull, PRs)

AI tools
├── Claude Code introduction (install, configure, use)
└── APIs and MCPs (keys, servers, integration)

Scaffolding strategy

  • Foundation: Step-by-step instructions with screenshots and expected output
  • Version control: Guided exercises with less hand-holding; participants make choices
  • AI tools: Goal-oriented tasks; participants choose how to accomplish them
  • Support reduction: Early tutorials say “type this command”; later tutorials say “figure out how to do X”

Tutorial template

Each tutorial follows this structure:

  1. Overview — What you’ll learn, prerequisites, estimated time
  2. Concepts — Brief explanation of what and why (not how yet)
  3. Walkthrough — Step-by-step worked example
  4. Practice — Independent exercises with hints
  5. Check your understanding — Self-assessment prompts
  6. Next steps — Links to related tutorials

Meeting facilitation

Session structure (60 minutes)

  • 0–5 min: Check-in, questions from last time
  • 5–15 min: Brief demo of the day’s topic
  • 15–45 min: Guided practice (participants follow along, then try independently)
  • 45–55 min: Open exploration and Q&A
  • 55–60 min: Recap, preview next session, quick feedback

Facilitation guidelines

  • Let participants struggle briefly before helping — productive struggle builds retention
  • Ask “what did you try?” before giving the answer
  • Encourage participants to help each other first
  • Keep demos short; get people typing as fast as possible
  • Have a backup plan if participants finish early (extension exercises) or get stuck (simpler version)

Managing mixed experience levels

  • Pair experienced participants with newer ones
  • Provide “fast track” notes for those who already know the basics
  • Offer extension challenges for advanced participants
  • Keep the main path accessible to the least experienced person in the room

Onboarding process

For new participants joining mid-stream

  1. Welcome email/message with link to the workgroup website
  2. Start here: Direct them to the tutorial index, starting with CLI basics
  3. Buddy assignment: Pair with a current participant
  4. First meeting: Arrive 10 minutes early for a quick orientation
  5. Catch-up plan: Work through tutorials at own pace; no need to match the group

Orientation checklist

Async learning support

Website as primary resource

The website is the canonical source of workgroup content. Meetings are supplements.

  • All topics covered in meetings must have corresponding tutorials on the site
  • Session notes posted within 48 hours of each meeting
  • Resources page kept current with links, cheatsheets, and troubleshooting

Communication channels

  • GitHub Issues: For questions, bug reports, and content requests
  • Email: For logistics and scheduling
  • In-person meetings: For hands-on practice and peer interaction

GitHub collaboration workflow

For the workgroup website

  • Main branch: Always deployable; content goes live via GitHub Pages
  • Feature branches: For drafting new tutorials or major changes
  • Pull requests: Used when participants contribute content
  • Issues: Track content requests, bugs, and feedback

For participant practice

  • Participants create their own repos during Git tutorials
  • Practice PRs and code review on a shared sandbox repo (if interest warrants)
  • Real collaboration: participants can contribute to the workgroup website

Technology integration

Tools used in sessions

  • Terminal.app / iTerm2: Primary interface for CLI work
  • Git: Version control (command-line, not GUI)
  • GitHub: Remote repos, collaboration, Pages deployment
  • Claude Code: Agentic AI assistant in the terminal
  • Text editor: VS Code recommended; any editor acceptable
  • Quarto: For authoring website content (participants may use this for their own work)

Tool installation sequence

  1. Terminal verification (built into macOS)
  2. Homebrew (macOS package manager)
  3. Git (via Homebrew or Xcode CLI tools)
  4. GitHub account + SSH key setup
  5. Claude Code (via npm or direct install)
  6. VS Code (optional, recommended)

Feedback and iteration

Feedback mechanisms

  • End-of-session check-in: One sentence on what worked and what didn’t
  • Periodic survey: Every 4–6 sessions, a short written survey
  • GitHub Issues: Open channel for suggestions and questions anytime
  • Informal conversation: The facilitator talks to participants

How feedback is used

  • Content adjusted based on what participants find confusing
  • Session pacing adjusted based on group progress
  • New topics added when requested
  • Tutorials revised when feedback identifies gaps

Quality assurance

Content review

  • Tutorials tested by running all commands on a clean macOS system
  • Code blocks verified for accuracy and copy-paste correctness
  • Links checked periodically for breakage
  • Participant feedback incorporated within one week

Website maintenance

  • Deploy after each content update via quarto publish gh-pages
  • Review site navigation quarterly
  • Archive old session notes (keep on site but deprioritize in navigation)
  • Keep resource links current

Risk management

Common challenges

  • Varied skill levels: Addressed through modular content and peer pairing
  • Irregular attendance: Addressed through async-first content design
  • Tool installation issues: Troubleshooting guide on the website; allocate session time for setup help
  • Participant frustration: Keep early content simple; celebrate small wins; normalize confusion

Sustainability

  • Website content persists even if meetings pause
  • Tutorials are reusable for future cohorts
  • Content is open-source and forkable
  • No dependency on a single facilitator for async content
